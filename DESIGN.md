## Problem Statement

The current game policy briefing bot (v1) is a single monolithic Python script (~1,700 lines) that works reliably — fetching RSS, filtering via Claude Haiku, summarizing in Korean, deduplicating via event keys, uploading to Notion, and emailing ~27 recipients at Pearl Abyss, Smilegate, Supercent, and law firms. It runs Mon/Wed/Fri via GitHub Actions. But it was built as a first vibe-coding project: monolithic, private repo, secrets in config.yaml, no web presence.

The user wants to rebuild this as a public, open-source, production-grade system in a new repository, keeping the original untouched. The v2 should be the first open-source game industry regulatory intelligence platform.

## What Makes This Cool

No open-source, game-industry-specific regulatory intelligence tool exists. Enterprise RegTech (CUBE, Regology, Compliance.ai) costs $50k-$500k+/year and targets banks and pharma. Game industry lawyers worldwide have no specialized, free, open tool for tracking regulatory changes across jurisdictions. This fills that gap.

The structured metadata approach means every article becomes a tagged node (jurisdiction, topic, regulatory phase, actors, event type), not just text. Over time, the system builds a navigable, queryable archive of global game regulation. Click "EU - Loot Box" and see every event across years. That turns a briefing newsletter into a public good. (Phase 2 may introduce actual graph relationships between nodes, e.g., "this Korean law was influenced by this EU directive.")

## Constraints

1. **Zero secrets in code.** Public repo means all email addresses, API keys, and recipient lists live in GitHub Secrets / environment variables only. config.yaml becomes a template.
2. **Near-zero infrastructure cost.** Static HTML on GitHub Pages (free), free GitHub Actions minutes, free Gmail SMTP. LLM API (Gemini 3.1 Flash-Lite) has per-call cost (~$0.25/1M input tokens, ~$1.50/1M output tokens), estimated at under $0.50/month based on v1 usage patterns (3 runs/week, ~10 articles/run).
3. **Original repo untouched.** New repo, new name, fresh git history. The v1 stays as-is in its private repo.
4. **Korean first, English in Phase 2.** Primary audience reads Korean. English summaries will be added in Phase 2 when the public site gains international visibility.
5. **Mon/Wed/Fri schedule.** Not daily. This affects archive density and eliminates the need for weekly recaps.

## Premises

1. The original repo stays untouched — new repo, fresh history, sentimental and practical safety net.
2. Public repo = zero secrets in code. All PII and credentials go to GitHub Secrets.
3. Static site on GitHub Pages (Jinja2 → HTML, no server, no database, no runtime cost).
4. Jurisdiction tracker is a stretch goal, not MVP. Core first: pipeline + web UI + email.
5. daily-brief's architecture is the reference implementation. Adapt patterns, don't copy code.
6. Weekly recaps are unnecessary at 3x/week frequency. A clean browsable archive is sufficient.

## Cross-Model Perspective

An independent Claude subagent reviewed this design cold. Key insights:

- **Coolest version not yet considered:** A living regulatory metadata store where each article is a tagged node with jurisdiction, topic, phase, and event type metadata. This enables "regulation timeline pages" (click "South Korea - Loot Box" to see every event), cross-jurisdiction pattern detection ("Japan is considering what Korea enacted 6 months ago"), and eventually per-topic RSS feeds (Phase 2).
- **What excites the user most:** "The public site becomes a shared resource." The subagent noted: "This isn't someone who wants a better internal tool. They're excited about owning the category."

- **Challenged no premises.** Validated the approach.

## Approaches Considered

### Approach A: daily-brief Clone (Modular Pipeline + Raw Jinja2)
Mirror daily-brief's architecture. Modular pipeline/, abstract LLM provider, Jinja2 → static HTML → GitHub Pages. Full control, no framework lock-in. Proven patterns. But more boilerplate for archive pagination and RSS feeds.

### Approach B: Pelican-Powered (Static Site Generator + Pipeline)
Use Pelican for site generation (archives, feeds, i18n for free), build only pipeline and metadata layers. Faster to reach browsable site, but adds a dependency with its own opinions. May fight custom design for game-law-specific views.

### Approach C: Structured Metadata First (Chosen)
Start with structured metadata as the core primitive. Every article produces a JSON node with jurisdiction, topic, regulatory phase, event_key. The site, email, and future jurisdiction tracker are all views on the same structured data. More upfront design on the metadata schema, but the jurisdiction tracker becomes a natural extension, not a bolt-on.

## Recommended Approach

**Approach C: Structured Metadata First.**

The user already extracts most of this metadata via event keys (jurisdiction, event_type, actors, object, action). Adding regulatory phase and game mechanic tags is a small prompt change. Starting with the data model means every future feature (jurisdiction tracker, topic timelines, per-topic RSS feeds) falls out naturally from the JSON layer.

### Data Model (BriefingNode)

```python
from dataclasses import dataclass
from enum import Enum

class Jurisdiction(str, Enum):
    EU = "EU"
    KR = "KR"
    US = "US"
    UK = "UK"
    JP = "JP"
    AU = "AU"
    CN = "CN"
    GLOBAL = "Global"

class RegulatoryPhase(str, Enum):
    PROPOSED = "proposed"           # Bill introduced, consultation announced
    PUBLIC_COMMENT = "public_comment"  # Open for public input
    ENACTED = "enacted"             # Law passed, regulation finalized
    ENFORCED = "enforced"           # Active enforcement action taken
    LITIGATION = "litigation"       # Court case filed or decided

class EventType(str, Enum):
    ENFORCEMENT = "enforcement"
    LEGISLATION = "legislation"
    LITIGATION = "litigation"
    POLICY = "policy"
    SECURITY_INCIDENT = "security_incident"
    BUSINESS = "business"
    OTHER = "other"

VALID_CATEGORIES = {
    "IP", "CONSUMER_MONETIZATION", "CONTENT_AGE", "PRIVACY_SECURITY",
    "PLATFORM_PUBLISHING", "AI_EMERGING", "MA_CORP_ANTITRUST",
    "ESPORTS_MARKETING", "LABOR_EMPLOYMENT", "ETC",
}

@dataclass
class LegalEvent:
    jurisdiction: Jurisdiction
    event_type: EventType
    regulatory_phase: RegulatoryPhase
    actors: list[str]          # ["Nintendo", "EU Commission"]
    object: str                # "Switch 2 battery design"
    action: str                # "compliance redesign"
    game_mechanic: str | None  # "loot_box", "age_rating", "data_collection", "ai_content", etc.
    time_hint: str             # "2027 deadline"

@dataclass
class BriefingNode:
    title: str
    url: str
    source: str
    pub_date: str
    category: str              # One of VALID_CATEGORIES
    summary_ko: list[str]      # 3-line summary in Korean
    event: LegalEvent
    event_key: str             # Hash for dedup across runs
    is_primary: bool           # True if this is the representative article for the event
    # Phase 2: summary_en: list[str]  — English summaries
```

### Retry & Rate Limiting

The abstract `LLMProvider` in `pipeline/llm/base.py` handles:
- **Retries:** `max_retries` (default: 2) with exponential backoff (1s, 2s)
- **Timeout:** `request_timeout_seconds` (default: 30) per LLM call
- **Rate limiting:** 429 responses trigger a 60-second wait before retry
- **JSON parsing:** Retry on malformed JSON with fallback regex extraction (carried from v1)

Config in `config.yaml`:
```yaml
llm:
  provider: gemini                    # gemini (default) or claude
  model: gemini-3.1-flash-lite        # default; claude: claude-haiku-4-5-20251001
  max_retries: 2
  request_timeout_seconds: 30
  max_input_chars: 8000
```

### Dedup Persistence

Cross-run deduplication uses `output/data/dedup_index.json`, a rolling index of event keys and URL hashes:

```json
{
  "schema_version": 1,
  "retention_days": 30,
  "entries": [
    {"event_key": "cf7fbde9388bf0b5", "url_hash": "a1b2c3...", "date": "2026-03-23"},
    ...
  ]
}
```

The index is pruned on each run (entries older than `retention_days` are dropped). At 3 runs/week with ~10 articles each, the index stays under ~400 entries. If the archive grows beyond 1000 entries, consider migrating to SQLite.

### Google Sheets Admin Log

`pipeline/admin/sheets.py` syncs each briefing run's results to a Google Sheet. This replaces v1's Notion integration with a lighter-weight admin interface.

**Flow:**
1. Pipeline produces BriefingNodes → JSON
2. `sheets.py` appends rows to the configured Google Sheet (one row per article)
3. Admin reviews in Google Sheets: edit summaries, delete irrelevant articles, flag issues
4. (Optional, Phase 2) Next pipeline run reads back edits from Sheets to update JSON

**Sheet columns:** `date | title | url | source | category | jurisdiction | phase | summary_ko | event_key | status`

The `status` column allows the admin to mark articles as `published` (default), `edited`, or `deleted`. The static site generator reads JSON as the source of truth for MVP; Sheets→JSON sync is a Phase 2 feature.

**Library:** `gspread` + Google service account credentials (stored in `GOOGLE_SHEETS_CREDENTIALS` secret).

### Graceful Degradation

| Component | If it fails | User sees |
|-----------|-------------|-----------|
| RSS feeds (tier_a) | Skip failed source | Fewer articles |
| RSS feeds (tier_b) | Expected, skip silently | No impact |
| LLM summarization | Use raw title + first paragraph | Unsummarized article |
| LLM classification | Default to "ETC" category, "Global" jurisdiction | Less structured metadata |
| Email delivery | Skip, log error | Site still deploys |
| Google Sheets sync | Skip, log error | Everything else works |
| GitHub Pages deploy | Retry once | Previous version stays live |

**Briefing always deploys.** Even if every API fails, the pipeline produces a page with whatever data succeeded.

### CSS Strategy

- **Web pages:** External stylesheet at `static/style.css`, linked in templates. Standard approach.
- **Email templates:** Inline CSS via `premailer` package at render time. Email clients strip `<link>` and `<style>` tags, so CSS must be inlined per-element for email delivery.

## Data Migration

V2 starts with a **clean slate**. The v1 Notion database and trend files remain in the private repo. Rationale: the v2 metadata schema (jurisdiction, regulatory_phase, game_mechanic) is richer than v1's output, so back-filling historical data would require re-processing all articles through the new prompts. This can be done as a one-time batch job after the v2 pipeline is stable, but is not a launch requirement.
